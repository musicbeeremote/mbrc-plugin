# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

MusicBee Remote (mbrc) is a plugin for MusicBee that allows remote control from the
Android and iOS client apps. The plugin runs a TCP server speaking a CRLF-delimited
JSON protocol.

The plugin core has been **rewritten in Rust** (`mbrc_core.dll`) and is called over FFI
from a thin C# shim (`mb_remote.dll`). The C# side is reduced to MusicBee's plugin entry
point plus the MusicBee API callback surface; the wire protocol, command dispatch, socket
server, discovery, caches, settings, and logging all live in Rust. The two DLLs ship
side-by-side in MusicBee's plugins folder.

The Rust core is **wire-compatible with the shipped C# plugin's V4 protocol** (byte-identical
frames, field order, and the iOS/Android client quirks preserved), so existing clients keep
working unchanged.

## Development Commands

### Building

`build.ps1` is the unified build (Rust core + C# plugin). Building requires Visual Studio
2026's MSBuild (2022 also works) for the C# side and the `i686-pc-windows-msvc` Rust target
for the core (both are x86 / 32-bit).

```powershell
.\build.ps1                       # both, Release
.\build.ps1 -Configuration Debug  # both, Debug (copies both DLLs to app\MusicBee\Plugins)
.\build.ps1 -Rust                 # just the Rust core (mbrc_core.dll)
.\build.ps1 -Plugin               # just the C# plugin (mb_remote.dll)
.\build.ps1 -Clean                # remove build output first
```

- Release builds the Rust core with the `plugin` profile (`panic = "unwind"`, so a Rust
  panic can never abort MusicBee).
- **Debug** copies both `mb_remote.dll` and `mbrc_core.dll` to `app\MusicBee\Plugins\` for a
  live test install; Release only stages them under `build\bin\plugin\`.
- `build-msbuild.ps1` builds only the C# solution (`MBRC.sln`) if you don't need the core.

### Formatting and lint

```bash
# Rust (run for the i686 target the plugin actually builds for)
cargo fmt --all
cargo clippy --workspace --all-targets --target i686-pc-windows-msvc -- -D warnings

# C#
dotnet format            # or: --verify-no-changes, whitespace, style, analyzers
```

### Tests

```bash
cargo test -p mbrc-core --target i686-pc-windows-msvc   # Rust core + integration suites
# C# xUnit suite: MusicBeeRemote.Core.Tests (built by MBRC.sln, sources under tests/csharp/)
```

### Development Environment
- **Rust core**: target `i686-pc-windows-msvc` (x86). Cargo workspace.
- **C# plugin**: .NET Framework 4.8, **max C# 7.3**, x86. IDE: Visual Studio 2026 (2022 also
  supported).

## Repository Layout

- `packages/mbrc-core/` - the Rust core (cdylib `mbrc_core.dll`): server, wire codec, command
  dispatch, discovery, caches, cover handling, FFI boundary, logging.
- `packages/mbrc-wire/`, `packages/mbrc-discovery/`, `packages/mbrc-capture/` - shared Rust crates
  (wire codec/handshake, UDP discovery, capture/fixture tooling).
- `packages/plugin/` - the C# plugin (`mb_remote.dll`): `Plugin.cs` entry point, `Ffi/` FFI shim
  + generated bindings, `Providers/` MusicBee API callbacks, `Settings/`, `Host/`.
- `packages/firewall-utility/` - elevated helper that adds the Windows firewall rule.
- `tools/mbrc-cli/` - the `mbrc` CLI (`send`, `monitor`, `capture`, `replay`, `trim`, ...); the
  read-only `monitor` subcommand is the validation harness.
- `tools/api-debugger/` - standalone Tauri + Vue API testing app (its own CI, not in `MBRC.sln`;
  use pnpm, not npm).
- `tests/golden/` - committed golden wire traces (`legacy-v4-{android,ios}.jsonl`).
- `MBRC.sln` - the C# projects (plugin, firewall-utility, MusicBeeRemote.Core.Tests). The Rust
  crates are a separate Cargo workspace (root `Cargo.toml`).

## Architecture Overview

### Communication flow
1. A client (Android/iOS) connects over TCP (port configurable, default 3000) and completes the
   `player` + `protocol` handshake.
2. Text protocol: `\r\n`-delimited JSON, `{"context":"<command>","data":<...>}`.
3. The Rust socket server parses each frame, dispatches by `context` to a command handler, and
   the handler reads/writes MusicBee state through FFI callbacks into C#.
4. Replies and event broadcasts are framed back to the client (`{"context":"<event>","data":<...>}`).

### The FFI boundary
- The Rust core is the source of truth for the C#↔Rust contract. Bindings are generated by
  **csbindgen** (`NativeBridge.Generated.cs` / enums) - **do not hand-edit generated files**;
  change the Rust `#[no_mangle]` surface and regenerate.
- FFI DTOs are serialized with **MessagePack** using the contractless resolver (property names
  as keys - match Rust serde field names). Rust->C# msgpack must use `to_vec_named`.
- **ILRepack** merges the C# core into `mb_remote.dll` with `Internalize=true`; an
  `InternalizeExclude` keeps the FFI DTO types public (MessagePack's dynamic formatter needs
  public types at runtime).

### Rust core responsibilities
- **Server** (`server/`): accept loop, per-connection tasks, session state machine, command
  dispatch, broadcaster, connection registry (per-IP/per-client caps, keepalive).
- **Wire** (`wire/`): V4 codec - byte-identical framing/field order, iOS vs Android variants,
  lenient parsing for iOS quirks.
- **Discovery** (`discovery.rs`): UDP multicast responder on `239.1.5.10:45345`.
- **Caches** (redb, `mbrc.redb`): library metadata cache + O(page) track paging index + album
  cover cache; a library fingerprint drives warm-persistence across restarts (skip rebuild when
  unchanged).
- **Settings**: `core_settings.json` is the single source of truth (Rust-owned); the MusicBee
  Configure panel edits it via FFI get/set.
- **Logging**: `mbrc-core.log` (UTC RFC3339 timestamps, runtime size-rotation with gzip,
  10 MiB cap x 3 generations). Wire frames log at DEBUG (list bodies capped to a sample +
  schema summary) / TRACE (full body); INFO is the default and logs no frames.

### Protocol
- **V4 only.** Pre-V4 clients are rejected at handshake. V4 is a maintained legacy target: its
  quirks are preserved byte-for-byte for compatibility with shipped Android (1.1.0-1.6.1) and iOS
  clients. See `docs/protocol.md` for the full command/event reference.
- Request: `{"context":"<command>","data":<...>}`; response/broadcast: `{"context":"<event>","data":<...>}`.

## Important Considerations

1. **x86 everywhere**: the C# plugin is x86/net48, so the Rust core must be `i686-pc-windows-msvc`.
2. **C# is a thin shim**: business logic lives in Rust; `Plugin.cs` is just MusicBee's entry point
   and the API callback surface. Prefer changing the Rust core over growing the C#.
3. **Two DLLs**: `mb_remote.dll` (managed) loads `mbrc_core.dll` (native) side-by-side; the native
   DLL is not embedded.
4. **Legacy C# constraint**: the plugin is limited to C# 7.3 (.NET Framework 4.8). The Rust core
   has no such limit.
5. **Wire compatibility is sacred**: V4 frames must stay byte-identical to the shipped plugin
   (field order included). Validate changes against `tests/golden/` and the `mbrc monitor` harness.
6. **Generated FFI code**: never hand-edit `packages/plugin/Ffi/Generated/*`; regenerate from Rust.
