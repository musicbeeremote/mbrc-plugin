# MusicBee Remote — Rust Core Architecture

Reference doc for how the Rust core (`mbrc-core/`) and the C# shim
(`plugin/`, `core/`) fit together. Companion to
`rust-core-remaining.md`, which tracks open work; this one captures
the design decisions that aren't going anywhere.

## Layout

```
┌─────────────────────────────────────────────────────────────┐
│                     MusicBee (C++)                          │
└─────────────────────────────────────────────────────────────┘
                            │ MusicBee Plugin API
                            ▼
┌─────────────────────────────────────────────────────────────┐
│                 mb_remote.dll (.NET Framework 4.8)          │
│       Plugin entrypoint + MusicBee API data providers       │
│       (Costura/ILRepack-merged: plugin/ + core/)            │
└─────────────────────────────────────────────────────────────┘
                            │ P/Invoke (C ABI)
                            ▼
┌─────────────────────────────────────────────────────────────┐
│                  mbrc_core.dll (Rust cdylib)                │
│   Tokio + Axum + Hyper. Owns: TCP server, legacy protocol,  │
│   HTTP/WS API, broadcast fan-out, UDP discovery, logging.   │
└─────────────────────────────────────────────────────────────┘
```

C# does what only C# can: implement the MusicBee plugin interface and
wrap the MusicBee API (~25 thread-pinned methods like `Player_PlayPause`,
`Library_QueryFiles`, `NowPlayingList_QueryReorder`). Rust does
everything else — networking, parsing, state, broadcast, scheduling.

## Why Rust?

The same shape is buildable in C#, Go, or Rust. Rust won on three axes
that matter for a plugin we ship inside someone else's process:

- **Binary size** — `mbrc_core.dll` is ~2.2 MB release. Go's runtime
  alone would be 11+ MB. Smaller is friendlier for installer + nightly
  zip + provenance attestation churn.
- **No GC** — MusicBee is a long-lived process and we run inside it;
  GC pauses on a managed runtime are someone else's problem.
- **`extern "C"` is free** — no CGO-style trampoline cost on every
  callback. We make hundreds of FFI hops per legacy session.

The tradeoffs (slower compile, smaller hiring pool than C#) are worth
the binary footprint and the predictable per-callback cost.

## Hybrid protocol server

A single `TcpListener` on the configured port (default 3000) accepts
both the legacy CRLF-delimited JSON protocol and the new HTTP/WS API.
We peek the first bytes off the stream and dispatch:

```rust
async fn classify(stream: &TcpStream) -> Protocol {
    let mut buf = [0u8; 16];
    let n = stream.peek(&mut buf).await?;
    match &buf[..n] {
        b if b.starts_with(b"GET ") || b.starts_with(b"POST ")
            || b.starts_with(b"PUT ") || b.starts_with(b"DELETE ")
            || b.starts_with(b"OPTIONS ") => Protocol::Http,
        b if b.starts_with(b"{") => Protocol::Legacy,
        _ => Protocol::Legacy, // safest fallback for the Android client
    }
}
```

HTTP/WS is served via `hyper_util::server::conn::auto` so the same
Axum stack handles both H1 and the WS upgrade. Legacy is its own
state machine (handshake → authenticated → command/broadcast loop).
Source: `mbrc-core/src/server/`.

### Legacy protocol — what we keep

V4 is a permanent compatibility branch. All its quirks (concatenated
keys, stringified `playervolume`, iOS `\'` escapes, paginated browse
envelopes the C# data providers don't know about) are preserved
*as-is* — we capture them in golden traces under
`mbrc-core/tests/golden/` and replay them in CI. Pre-V4 is rejected
at handshake with a `notallowed` frame and a human-readable reason.

A clean-slate **v5** is intended to live next to v4, not replace it.
That's what the HTTP/WS API is.

### HTTP/WS — the long-lived contract

`/api/v1/*` is the embedded webapp's API. Same data as legacy, cleaner
JSON (no `{context, data}` envelope), CORS-enabled for local
development. WebSocket multiplexes broadcast events. The exact route
shape isn't pinned yet — see `rust-core-remaining.md` §4. Source:
`mbrc-core/src/server/api/`.

## C ABI

The FFI surface is **the source of truth on the Rust side**. C#
bindings (`plugin/Services/Generated/NativeBridge.Generated.cs`) are
generated by csbindgen at `cargo build` time; CI fails if they're
stale. Don't hand-edit the generated file.

For canonical signatures, read:

- `mbrc-core/src/lib.rs` — `extern "C"` exports (`mbrc_init`,
  `mbrc_handle_notification`, `mbrc_log`, `mbrc_set_log_level`, etc.).
- `mbrc-core/src/ffi/types.rs` — the `MbrcCallbacks` struct,
  `QueryType` and `CommandType` enums.

### Two-tier callback design

`MbrcCallbacks` (passed once at `mbrc_init`) splits into:

- **Thin callbacks** — direct function pointers for latency-sensitive
  transport actions (`player_play_pause`, `player_stop`, `player_next`,
  `player_previous`, `player_set_volume`, `player_set_position`). One
  FFI hop, no marshalling.
- **Fat callbacks** — `query_data(QueryType, msgpack_params) → msgpack_result`
  and `execute_command(CommandType, msgpack_params) → i32`. Everything
  else flows through these. The numeric `QueryType` / `CommandType`
  enums are the contract; payloads are MessagePack-encoded DTOs.

The fat-callback path lets us add new query/command variants by
extending the enums + DTOs without changing the FFI struct layout —
which is what kept the C ABI stable across W2's full surface
expansion.

`mbrc_log` is a separate one-way export (C# → Rust) that pipes
log lines into the Rust `tracing` subscriber, so the plugin only
writes one log file. See "Logging" below.

### MessagePack drift guard

DTO field names must match exactly between Rust serde and C#
contractless MessagePack. Two safety nets:

- **24 canonical fixtures** under `mbrc-core/tests/schemas/`. The
  Rust integration test deserialises each into the matching struct;
  a sibling C# xunit test
  (`tests/Serialization/NativeDtoDriftTests.cs`) does the same on
  its side. Either side renaming a field fails CI.
- **`rmp_serde::to_vec_named`, never `to_vec`**. The contractless
  resolver on C# reads structs as MessagePack maps; positional arrays
  blow up at deserialise time. A regression here surfaces only at
  runtime, hence the convention is documented at every callsite.

## Threading

```
                Tokio multi-thread runtime (mbrc-core)
┌──────────────────────────────────────────────────────────────┐
│  TCP accept · UDP discovery · ping timer                     │
│        │                                                     │
│        ▼                                                     │
│  Per-connection task (legacy state machine OR Axum handler)  │
│        │                                                     │
│        ▼                                                     │
│  spawn_blocking ───────────────► C# callback (MusicBee API)  │
└──────────────────────────────────────────────────────────────┘
```

MusicBee's API is **not thread-safe**: most methods must run on the
thread MusicBee called us from, or at least serially. Async tasks
that need data go through `tokio::task::spawn_blocking` so we hop
onto a blocking-pool worker before we hand off to C#. The C# side
serialises FFI calls via locking inside the data providers where
needed.

`SafeCallbacks` (`mbrc-core/src/ffi/callbacks.rs`) wraps the raw
function pointers with typed Rust methods (`set_repeat(&str)`,
`query_no_params::<T>(QueryType)`, etc.) so handler code never
touches `extern "C"` directly.

The C# side keeps:

- `plugin/Plugin.cs` — the actual MusicBee plugin entry point
- `plugin/Services/NativeBridge.cs` — FFI lifecycle, callback dispatch,
  MessagePack DTO marshalling
- `plugin/DataProviders/` — wrappers over the MusicBee API that the
  Rust callbacks call into
- `core/` — DI module, settings, cover cache, MusicBee API adapters,
  WinForms settings UI, the logging shims that talk to Rust

## Configuration transport

C# never tells Rust the listen port at runtime. Instead, Rust reads
`{storage}/mb_remote/core_settings.json` at init (and on
`CoreRestartRequestedEvent`). Source: `mbrc-core/src/config.rs`. The
C# settings UI writes to the same JSON file, then publishes the
restart event; `Plugin.cs` calls `mbrc_stop_networking` →
150 ms → `mbrc_start_networking` to re-read.

This avoids a richer settings RPC at the FFI boundary and means the
JSON file is the *only* config surface — same source of truth for the
plugin and any future external tools.

## Logging

One file: `{storage}/mb_remote/mbrc.log`, daily rotation, owned by
the Rust `tracing` subscriber.

- Rust events go directly to the file via `tracing-appender`.
- C# events flow through `mbrc_log(level, target, message)` into the
  same subscriber. Rendered output sits in the same file, same
  timeline, same format. (`core/Infrastructure/Logging/Implementations/RustLogBridge.cs`.)
- `mbrc_set_log_level(directive)` reconfigures the active `EnvFilter`
  at runtime — used by the settings-UI debug toggle.
- The C# side falls back to `BootstrapLogger.cs` (a tiny file writer
  at `bootstrap.log`) when `mbrc_log` returns -1. That covers the
  brief windows before `mbrc_init` returns and after `mbrc_shutdown`,
  plus catastrophic FFI failures (DLL missing, wrong arch).

NLog is gone. The `IPluginLogger` interface still exists, but the
implementation is the FFI bridge.

## Backwards compatibility

| Surface | Guarantee |
|---|---|
| Legacy V4 protocol | byte-compatible — golden-trace tested, all V4 quirks preserved |
| Pre-V4 protocols | rejected at handshake with a `notallowed` reason |
| Default port | 3000 (unchanged) |
| UDP discovery | `239.1.5.10:45345`, same JSON shape as the C# server |
| MessagePack DTOs | drift-guarded by fixtures; renaming a field fails CI |

## Build pipeline

`dotnet build` runs `build-rust.ps1` first via the `BuildRustCore`
MSBuild target (`plugin/mbrc-plugin.csproj`), then C#. Both DLLs land
in the plugin output. CI runs the Rust job first, fails on
`cargo fmt`/`cargo clippy -- -D warnings`/`cargo test`, then builds
release for `i686-pc-windows-msvc`, then the C# job consumes the
artifact.

`mbrc_core.dll` is shipped alongside `mb_remote.dll` in the release
zip and the NSIS installer, with both included in the build
provenance attestation.
